# vibeCoding 项目系统化测试工程方向与开发方案

## 1. 项目定位

本工程不是为某一个具体项目定制的测试脚本集合，而是一个通用的 vibeCoding 项目测试平台。

它的目标是：对由 vibeCoding 方式生成或快速迭代出来的软件项目进行系统化、可复用、可扩展的需求验收，优先发现项目是否满足需求文档、是否存在功能缺失、实现偏差和明显交付风险。

本工程默认不按传统人工项目的代码审美进行评价。面对 90% 以上由 AI 生成的项目时，代码是否优雅、结构是否清晰、抽象是否漂亮不是主要价值。默认判断权重为：90% 用于功能是否符合需求文档，10% 用于 key 明码、明显 hard-code、固定样例过拟合、入口不可运行、Web UI 明显不可操作等交付风险。安全性、健壮性、可维护性和架构质量只在明显影响功能验收或交付事故时进入主要结论。

上级目录中的 `github-deepSearch` 可以作为早期样例项目，用于验证测试流程和报告能力，但不能让本工程的架构、目录结构、规则体系或测试逻辑绑定到该项目。

### 1.1 核心原则：自举测试与自身达标

本工程必须首先成为一个优秀 vibeCoding 项目的样板。它不能只用于测试其他项目，也必须持续用自身定义的测试标准测试自己。

这一原则要求：

- 本工程对外宣称支持的每一类测试能力，都应优先或同步用于测试本工程自身。
- 本工程的文档、代码、测试、报告、成本控制、安全隔离和 CI 质量门禁，都必须满足本工程定义的优秀 vibeCoding 项目标准。
- 如果某个标准本工程自身尚未达到，必须在报告和路线图中明确标注为未完成能力，而不能把它作为已经成熟的外部评判标准。
- 后续每次迭代都应运行本工程的自测流程，并保存报告作为质量证据。
- 测试平台自己无法通过的标准，不应被用来苛责其他被测项目。

自举测试采用分阶段推进：

- MVP 阶段：项目至少能通过基础文档检查、基础 CLI 可运行性、基础项目识别、基础报告生成。
- Standard 阶段：项目能通过自身的需求抽取、profile 匹配、工具执行、报告 JSON、成本统计等标准测试。
- Deep/Audit 阶段：项目能通过自身的隔离执行、浏览器自动化、API 验证、安全扫描、覆盖矩阵和历史报告对比。

## 2. 为什么需要这个工程

直接使用 Codex、Claude Code 或其他 coding agent 对源码进行检查是有价值的，但它更像一次性的智能审查。本工程的价值是把这种能力工程化、流程化、可复现化，并与成熟测试工具结合。

相比直接让 coding agent 看源码，本工程的优势包括：

- 可复现：每次测试有固定输入、执行计划、工具版本、日志、截图和报告，而不是只依赖一次对话判断。
- 可比较：同一个项目多次测试可以比较历史结果，不同项目也可以用统一标准横向比较。
- 可自动化：可以在 CI、本地批处理或定时任务中运行，而不是人工每次重新描述需求。
- 证据更完整：不仅给出“代码看起来有问题”，还保存构建日志、测试输出、浏览器截图、接口响应、覆盖率、安全扫描结果等证据。
- 覆盖更全面：LLM 负责需求抽取、语义比对和测试生成，开源工具负责静态分析、安全扫描、浏览器自动化、性能压测、覆盖率等确定性任务。
- 更适合 vibeCoding：vibeCoding 项目经常只有汇总 Markdown，本工程把“从非标准文档抽取需求，再映射到测试计划”作为核心能力。
- 质量门禁更清晰：可以把问题分级、设定通过标准，并输出机器可读的 JSON，方便后续自动化处理。

因此，本工程不是替代 Codex 或 Claude Code，而是把 LLM 作为测试平台中的一个能力模块，让它和传统测试工具一起工作。

## 3. 成本与依赖

本工程的成本需要在设计阶段明确，因为深度测试不是零成本。

### 3.1 单次执行成本估算

单次测试成本取决于项目规模、测试深度、是否启用 LLM、是否需要浏览器自动化、是否需要启动数据库或第三方服务。

建议把执行模式分为四档：

| 模式 | 适用场景 | 主要内容 | 典型成本 |
| --- | --- | --- | --- |
| `basic` | 快速初筛 | 克隆、项目识别、文档读取、lint/test/build 中可用项、安全和密钥扫描 | 本地运行基本为 0 美元，主要是机器时间 |
| `standard` | 常规验收 | basic + LLM 需求抽取 + 报告总结 + 基础测试数据生成 | 小项目通常低于 1 美元，中型项目约 1-5 美元 |
| `deep` | 深度验收 | standard + 浏览器自动化/API 测试生成 + 失败归因 + 覆盖矩阵 | 中型项目约 5-20 美元，大项目更高 |
| `audit` | 发布前审计 | deep + 多模型复核 + 性能/安全专项 + 更长运行时间 | 可能超过 20 美元，应设置预算上限 |

这里的金额主要指 LLM API 成本。开源工具本身通常免费，但会消耗本地或 CI 计算资源。

粗略估算公式：

```text
单次成本 = LLM token 成本 + 可选商业工具成本 + 计算/CI 成本 + 存储成本
```

如果使用 OpenAI API，成本可按官方价格中的每百万 input/output token 单价估算。以 2026 年 6 月可见的 OpenAI API 标准价格为例，`GPT-5.4 mini` 为 input 0.75 美元/百万 token、output 4.50 美元/百万 token，`GPT-5.4` 为 input 2.50 美元/百万 token、output 15.00 美元/百万 token，`GPT-5.5` 为 input 5.00 美元/百万 token、output 30.00 美元/百万 token。实际价格应以供应商当前价格页为准。

示例估算：

| LLM 用量 | mini 级模型估算 | 大模型估算 | 典型场景 |
| --- | --- | --- | --- |
| 50k input + 10k output | 约 0.08 美元 | 约 0.28-0.55 美元 | 小项目摘要和基础报告 |
| 200k input + 50k output | 约 0.38 美元 | 约 1.25-2.50 美元 | 中型项目需求抽取和报告 |
| 1M input + 200k output | 约 1.65 美元 | 约 5.50-11.00 美元 | 大项目深度分析 |

工程实现上必须支持预算控制：

- 设置单次最大金额，例如 `max_cost_usd`。
- 设置最大 token 数。
- 设置最大文件数和最大代码体积。
- 对依赖目录、构建产物、图片、锁文件等进行默认排除。
- 优先摘要后分析，避免把整个仓库一次性送入 LLM。
- 对相同文件摘要做缓存。
- 在达到预算前降级为确定性工具测试。

### 3.2 LLM 成本

如果启用需求抽取、语义比对、测试用例生成、报告总结等智能能力，通常需要配置 LLM API key。

可能的成本来源：

- OpenAI、Anthropic、Google、DeepSeek、通义千问等模型 API 调用费用。
- 长文档和大代码库分析产生的 token 成本。
- 多轮测试生成和失败分析产生的额外调用成本。
- 使用多模型交叉验证时的额外成本。

早期设计讨论过确定性基础扫描、LLM 辅助扫描和深度 LLM 分析三类路径。

当前实现已经收敛为单一 LLM-required acceptance 路径：每次 `scan` 都需要 LLM API key，用于生成项目专属验收契约，并在报告中记录 preflight token/cost 估算和实际使用量。

### 3.3 工具与运行成本

除 LLM 外，还存在以下成本：

- 安装依赖和构建项目的时间成本。
- 浏览器自动化、端到端测试、性能压测的运行时间成本。
- Docker、数据库、缓存、消息队列等测试环境成本。
- 安全扫描和依赖扫描的计算成本。
- 截图、日志、覆盖率、报告等 artifacts 的存储成本。
- 某些商业工具或云服务的授权成本，例如 SonarCloud、Snyk、BrowserStack。

### 3.4 维护成本

工程长期运行还需要维护：

- 不同语言和框架的适配器。
- 测试 profile。
- 开源工具版本兼容。
- 误报和漏报规则。
- 报告模板。
- CI 集成。

早期应优先选择开源、CLI 友好、可本地运行的工具，避免让项目一开始就依赖高成本商业服务。

## 4. 支持的测试方式

本工程需要同时支持确定性工具测试、运行时行为测试、LLM 辅助测试和人工可复核报告。

### 4.1 静态测试

- 代码 lint。
- 类型检查。
- 格式检查。
- 依赖漏洞扫描。
- 密钥泄露扫描。
- SAST 安全规则扫描。
- 复杂度、重复代码、大文件、大函数检测。
- 文档和配置一致性检查。

### 4.2 单元、集成和构建测试

- 自动发现并运行项目已有测试。
- 执行构建命令。
- 收集覆盖率。
- 检查测试失败、跳过、超时和不稳定测试。
- 在缺少测试时生成风险提示。

### 4.3 浏览器自动化测试

Web 项目需要支持浏览器自动化。优先使用 Playwright。

可支持能力：

- 自动打开页面。
- 模拟鼠标点击。
- 模拟键盘输入。
- 表单填写。
- 页面导航。
- 文件上传。
- 多视口测试，例如桌面端和移动端。
- 截图和视频记录。
- 控制台错误收集。
- 网络请求失败收集。
- 可访问性扫描。
- 视觉回归测试。

浏览器自动化既可以执行预设 smoke test，也可以根据需求文档和页面结构生成候选测试流程。

### 4.4 API 测试

- 发现 OpenAPI、Swagger、Postman collection、路由文件和请求示例。
- 检查接口可达性。
- 校验状态码、响应结构、错误响应和鉴权行为。
- 支持 schema fuzzing。
- 支持契约测试。
- 可集成 Newman、Schemathesis、Dredd、REST Assured、k6。

### 4.5 CLI 测试

- 检查 help 和 version。
- 测试必填参数、非法参数、退出码。
- 使用示例输入验证输出。
- 检查文件输出、副作用和错误提示。
- 支持快照测试和 golden file 测试。

### 4.6 测试数据生成

本工程应支持测试数据生成，但必须可控、可复现。

支持方向：

- 基于 schema 生成数据。
- 基于示例生成相似数据。
- 生成边界值，例如空值、超长字符串、特殊字符、非法枚举、极大数、极小数。
- 生成安全测试 payload，例如 XSS、SQL 注入、路径穿越。
- 生成多语言文本和特殊编码输入。
- 生成 API 请求组合。
- 生成 CLI 参数组合。
- 为 AI 应用生成 prompt injection 和越权访问测试样本。

生成的数据需要记录 seed、规则来源和用途，避免测试不可复现。

### 4.7 性能和稳定性测试

- Web/API 轻量压测。
- 页面性能和 Lighthouse 检查。
- 接口响应时间检查。
- 并发和超时测试。
- 长任务稳定性测试。
- 内存增长和资源占用观察。

### 4.8 LLM 辅助测试

LLM 适合负责：

- 从 README、汇总 Markdown、issue、聊天记录中抽取需求。
- 把模糊描述转为候选验收点。
- 根据项目类型生成测试计划。
- 根据失败日志给出归因建议。
- 根据页面结构或 API schema 生成候选测试用例。
- 将工具输出整理为人类可读报告。

LLM 不应单独作为最终判断依据。所有重要结论应尽量绑定到可执行测试、代码位置、日志、截图或扫描结果。

## 5. 测试边界与不可发现问题

本工程必须明确边界，不能承诺“发现所有问题”。

### 5.1 难以通过自动测试发现的问题

以下问题通常无法完全依靠自动测试发现：

- 需求本身错误或产品方向错误。
- 文档没有描述、代码也没有明显线索的隐含业务规则。
- 需要真实用户长期使用才能暴露的体验问题。
- 需要真实生产数据才能判断的业务质量问题。
- 第三方服务在线上环境中的偶发问题。
- 复杂权限组织结构下的真实治理问题。
- 长周期并发、资源泄漏或数据漂移问题。
- 只有特定硬件、地区、网络、浏览器插件或账号状态下才出现的问题。
- 视觉审美、品牌调性、内容准确性等主观质量问题。
- LLM 应用中的所有越狱和 prompt injection 变体。
- 恶意攻击者通过未知漏洞链组合出的高级安全问题。

### 5.2 测试结论的表达边界

报告中不应使用“项目完全没有问题”这类绝对表述，而应使用：

- 在当前输入文档范围内，哪些需求被验证通过。
- 在当前测试环境内，哪些行为被验证通过。
- 哪些需求未验证。
- 哪些需求缺少足够信息。
- 哪些问题是确定失败。
- 哪些问题是风险提示。
- 哪些结论需要人工确认。

### 5.3 需要人工参与的判断

以下内容需要人工确认：

- 需求优先级。
- 模糊需求的真实意图。
- 产品体验是否符合目标用户预期。
- 业务规则是否符合实际业务。
- 安全风险是否接受。
- 性能阈值是否合理。
- 测试报告中的误报和可接受风险。

本工程的正确定位是“增强测试和验收能力”，不是取消人工验收。

## 6. 待测项目需要提供的内容

为了让测试足够深入，理想情况下，待测 vibeCoding 项目应包含以下内容。

### 6.1 只提供 GitHub 链接的情况

用户只提供一个 GitHub 仓库链接时，本工程应可以直接开始测试，但测试深度取决于仓库本身的信息完整度和运行条件。

支持流程：

1. 克隆仓库到临时工作区。
2. 识别默认分支、项目语言、框架、包管理器和目录结构。
3. 搜索 README、docs、Markdown、OpenAPI、package 配置、测试配置和启动脚本。
4. 抽取需求候选项和可执行命令。
5. 选择匹配的测试 profile。
6. 在隔离环境中安装依赖、构建、运行测试和扫描。
7. 如果识别出 Web/API/CLI 入口，则尝试启动服务并执行行为测试。
8. 输出报告，并明确哪些内容因缺少配置或密钥无法验证。

可直接测试的条件：

- 仓库是公开可访问的，或用户提供了访问凭证。
- 项目有基本文档或可识别的入口文件。
- 依赖可以安装。
- 构建和启动不依赖缺失的私有服务。
- 不需要未提供的生产密钥、账号或数据库。

只能部分测试的情况：

- 私有仓库但未提供 token。
- 仓库缺少 README、使用说明或入口文件。
- 依赖私有包源但未提供凭证。
- 需要外部数据库、第三方 API、云服务或真实账号。
- 需要复杂人工配置才能启动。
- 项目包含多个子项目但没有说明测试目标。

安全要求：

- 不应在宿主机上直接执行陌生仓库的动态代码。
- 默认使用临时目录、容器、受限权限和网络策略。
- 对安装脚本、postinstall、迁移脚本、shell 脚本等高风险行为进行记录和限制。
- 默认不向被测项目注入用户的真实密钥，除非用户明确配置。
- 报告中需要标注执行了哪些命令、产生了哪些副作用、哪些步骤被安全策略阻止。

### 6.2 最低输入

最低只要求：

- 项目源码目录。
- 至少一份描述项目目标、功能或使用方式的文档。

很多 vibeCoding 项目只有一个汇总 Markdown，这种情况必须作为核心支持场景。该 Markdown 不一定是正式需求文档，可能混合了背景、功能、实现说明、使用方式、限制、后续计划和示例。

### 6.3 推荐输入

如果项目具备以下内容，测试质量会更高：

- README 或项目说明。
- 需求汇总 Markdown。
- API 文档或接口示例。
- UI 截图、原型或页面说明。
- `.env.example` 或配置说明。
- 启动命令、构建命令、测试命令。
- 示例输入输出。
- 部署说明。
- 已有测试用例。
- 约束条件，例如性能目标、权限模型、数据范围、浏览器兼容性、运行环境。

### 6.4 文档解析原则

测试工程不能假设待测项目一定有标准 PRD。它需要从现有文档中抽取：

- 项目目标。
- 用户角色。
- 核心功能。
- 输入输出。
- 使用流程。
- 数据模型。
- 外部依赖。
- 明确约束。
- 隐含验收标准。
- 未完成事项。
- 风险或模糊点。

抽取结果应被保存为中间产物，供后续测试计划和最终报告引用。

## 7. 总体架构

建议采用“核心编排器 + 项目适配器 + 工具插件 + 报告生成器”的结构。

```text
agent-test
├── core                 # 测试编排、任务调度、结果模型
├── analyzers            # 项目识别、需求抽取、代码结构分析
├── adapters             # 不同语言和项目类型的适配器
├── tools                # 对开源工具的封装
├── profiles             # 测试策略配置，例如 web、api、cli、library
├── rules                # 通用规则和可配置质量门禁
├── reports              # 报告模板和生成逻辑
├── artifacts            # 日志、截图、覆盖率和扫描结果
├── examples             # 示例配置和样例项目说明
└── docs                 # 工程自身文档
```

核心设计原则：

- core 不包含具体项目规则。
- adapters 负责识别和执行不同技术栈的标准命令。
- tools 只封装外部工具，不把结果直接解释成最终结论。
- reports 负责把需求、执行结果和证据关联起来。
- examples 可以包含 `github-deepSearch` 这类样例，但不能污染 core。

## 8. 标准测试流程

1. 项目摄取  
   输入待测项目路径，读取文件树、包管理文件、配置文件、文档和入口文件。

2. 项目识别  
   判断语言、框架、项目类型、运行方式和测试能力。

3. 需求抽取  
   从 README、汇总 Markdown、docs、注释和示例中生成需求清单。

4. 测试计划生成  
   根据项目类型和需求清单选择测试 profile，并生成可执行任务。

5. 测试数据准备  
   根据 schema、示例、边界条件和安全规则生成可复现测试数据。

6. 工具执行  
   运行构建、测试、类型检查、lint、安全扫描、依赖检查、覆盖率、端到端测试等。

7. 行为验证  
   对 CLI/API/Web 项目执行黑盒或半黑盒测试，记录输入、输出、截图、响应和日志。

8. 语义比对  
   将需求清单与实际功能、代码路径、测试结果和运行证据进行对照。

9. 报告生成  
   输出 Markdown/HTML/JSON 报告，包含结论、问题、证据、严重程度、建议和可复现步骤。

## 9. 测试维度

### 9.1 需求符合性

检查项目是否实现文档中承诺的功能：

- 功能是否存在。
- 功能入口是否可用。
- 输入输出是否符合描述。
- 边界条件是否处理。
- 错误场景是否有合理反馈。
- 文档描述与实际行为是否一致。

### 9.2 可运行性

检查项目是否能被正常安装、启动、构建和测试：

- 依赖安装是否成功。
- 启动命令是否明确。
- 构建是否通过。
- 默认配置是否完整。
- 示例命令是否可运行。
- 新环境是否能复现。

### 9.3 静态质量

根据语言和技术栈接入成熟工具：

- JavaScript/TypeScript：ESLint、TypeScript、Biome、Vitest、Jest、Playwright。
- Python：ruff、mypy、pytest、coverage.py、bandit。
- Go：go test、go vet、staticcheck、gosec。
- Rust：cargo test、clippy、fmt、cargo-audit。
- Java：Maven/Gradle test、SpotBugs、Checkstyle、PMD。
- 通用：Semgrep、SonarQube/SonarCloud、CodeQL。

### 9.4 安全风险

覆盖常见风险：

- 依赖漏洞。
- 硬编码密钥。
- 注入风险。
- SSRF、XSS、CSRF。
- 不安全文件读写。
- 权限绕过。
- 不安全反序列化。
- 敏感日志输出。
- AI 应用中的 prompt injection 和数据泄露风险。

可集成工具：

- Semgrep。
- Gitleaks。
- TruffleHog。
- npm audit / pnpm audit / yarn audit。
- pip-audit / safety。
- cargo audit。
- osv-scanner。
- OWASP ZAP。

### 9.5 API 测试

对 API 项目或全栈项目：

- 自动发现 OpenAPI、Swagger、路由文件和请求示例。
- 生成接口可达性测试。
- 校验状态码、响应结构、错误响应和鉴权行为。
- 支持 Postman/Newman、Schemathesis、Dredd、REST Assured 等工具。

### 9.6 Web UI 测试

对 Web 项目：

- 使用 Playwright 进行页面启动、导航、核心流程、表单、错误态和截图检查。
- 检查移动端和桌面端基础布局。
- 检查控制台错误和网络失败。
- 接入 axe-core 做无障碍扫描。
- 对关键页面保存截图作为报告证据。

### 9.7 CLI 测试

对命令行工具：

- 检查 help/version。
- 检查必填参数、非法参数、退出码。
- 使用示例输入验证输出。
- 检查文件输出、副作用和错误提示。

### 9.8 性能与稳定性

根据项目类型选择轻量或深入测试：

- Web/API：k6、autocannon、wrk、Locust。
- 前端：Lighthouse、bundle analyzer。
- 后端：响应时间、并发、资源占用。
- 长任务：超时、重试、内存增长、异常恢复。

### 9.9 可维护性

检查：

- 项目结构是否清晰。
- 核心模块是否过度耦合。
- 是否存在大文件、大函数、重复代码。
- 配置和业务逻辑是否混杂。
- 是否有测试入口。
- 是否有持续集成配置。
- 是否存在过期 TODO 或明显未完成代码。

### 9.10 文档一致性

检查：

- README 中的安装、启动、测试命令是否有效。
- 环境变量是否说明完整。
- 示例是否能复现。
- 文档承诺和代码能力是否一致。
- 已知限制是否清楚。

## 10. 测试 Profile

为了保持通用性，本工程应使用 profile 描述不同项目的测试策略。

示例：

```yaml
name: web-ts
match:
  files:
    - package.json
  dependencies:
    - react
    - vue
    - next
    - vite
steps:
  - install
  - lint
  - typecheck
  - unit-test
  - build
  - playwright-smoke
  - accessibility
  - dependency-audit
  - secret-scan
```

Profile 不应写死具体项目名称，而应基于语言、框架、文件特征和用户配置匹配。

## 11. 报告格式

最终报告至少包含：

- 项目基本信息。
- 识别到的技术栈。
- 需求清单。
- 测试范围。
- 执行环境。
- 成本摘要，例如 LLM 调用次数、token 用量、耗时。
- 总体结论。
- 通过项。
- 失败项。
- 未验证项。
- 需求不满足项。
- 工程质量问题。
- 安全问题。
- 性能问题。
- 文档问题。
- 严重程度。
- 复现步骤。
- 证据链接，例如日志、截图、测试输出。
- 修复建议。
- 测试边界和未覆盖风险。

当前实现应同时输出：

- `AGENT_TEST_QA_REPORT.md`：适合开发人员、测试 agent 和修复 agent 阅读。
- `USER_QA_SUMMARY.pdf`：适合非技术用户、产品或业务负责人直接打开阅读。
- `report.json`：适合自动化系统消费。
- `artifacts/`：保存日志、截图、覆盖率、扫描结果。

## 12. 问题分级

建议采用以下等级：

- `Blocker`：项目无法安装、无法启动、核心功能完全不可用、存在高危安全问题。
- `Critical`：核心需求严重不符合，主要流程中断，数据安全或权限模型明显错误。
- `Major`：重要功能缺失、边界条件失败、构建或测试不稳定。
- `Minor`：局部体验、文档、错误提示、低风险实现问题。
- `Info`：建议优化项或需要人工确认的观察结果。

## 13. 开发阶段规划

### 阶段一：最小可用版本

目标是跑通一个通用测试闭环：

- 支持输入项目路径。
- 自动识别 Node/Python 基础项目。
- 提取 README/Markdown 中的需求候选项。
- 执行安装、构建、测试、lint、安全扫描中的可用项。
- 支持 LLM-required acceptance 路径。
- 生成开发/agent Markdown 报告、纯用户 PDF 摘要和 JSON 机器报告。
- 使用一个或多个样例项目验证流程。

### 阶段二：多测试方式扩展

- 增加 Go、Rust、Java 等适配器。
- 增加 Web/API/CLI profile。
- 增加 Playwright、Schemathesis、Semgrep、Gitleaks 等集成。
- 增加测试数据生成。
- 增加 JSON 报告。
- 增加 artifacts 管理。

### 阶段三：需求到实现的深度验证

- 更稳定地从汇总 Markdown 中抽取需求。
- 建立需求覆盖矩阵。
- 将测试结果映射到具体需求。
- 支持自动生成部分黑盒测试。
- 支持人工确认模糊需求。

### 阶段四：平台化和 CI 集成

- 支持配置文件。
- 支持 GitHub Actions/GitLab CI。
- 支持质量门禁。
- 支持历史报告对比。
- 支持插件化规则。
- 支持成本统计和预算限制。

## 15. 与具体样例项目的关系

`github-deepSearch` 可作为样例被测项目，用来验证：

- 汇总 Markdown 需求抽取是否有效。
- Node 或 Web 项目识别是否准确。
- 安装、构建、测试、运行流程是否可被自动发现。
- 浏览器自动化和 API 测试是否能产出证据。
- 最终报告是否能指出需求偏差和工程问题。

但本工程中不应出现只服务于 `github-deepSearch` 的核心逻辑。若确实需要项目特定规则，应放入 examples 或 fixtures，不进入 core。

## 16. 成功标准

本工程的成功不是“跑了多少工具”，而是能否稳定回答以下问题：

1. 这个项目声称要做什么？
2. 它实际做到了什么？
3. 哪些需求没有做到？
4. 哪些地方虽然能运行但存在明显质量、安全或交付风险？
5. 问题证据在哪里？
6. 修复优先级是什么？
7. 本次测试花费了多少成本？
8. 哪些问题无法通过当前测试发现？
9. 同样的方法能否迁移到另一个 vibeCoding 项目？
10. 本工程自身是否通过了同等级别的自举测试？

当本工程能对多个不同类型的 vibeCoding 项目给出一致、可复现、有证据、有边界说明的评估报告时，方向就是正确的。

### 16.1 人工测试基本水平是通用门槛

本工程面对的不是传统人工项目，而是 90% 以上由 AI 生成或低人工介入生成的 vibeCoding 项目。对这类项目，最常见也最有价值的问题往往出现在用户最终产物中：功能看似完成，但输出空白、无关、重复、无法复核，或者把不合格候选包装成可用结果。

因此，所有项目类型都必须满足同一条基础标准：`agent-test` 产出的报告至少要覆盖人工测试人员会检查的用户基本流程，并给出同等质量的缺陷交付物。

这意味着：

- 测试不能只证明命令、页面或接口“跑过”。
- 测试必须检查用户实际看到的输出是否满足需求。
- 搜索、推荐、报告、生成、分析类项目必须检查输出内容质量。
- Web 项目必须检查浏览器真实流程和用户可见结果。
- CLI/API 项目必须检查命令或接口返回给用户/调用方的最终内容。
- 如果报告引用外部项目、仓库、网站或文件，必须尽可能打开或核验其基本相关性。
- 如果这类检查没有执行，最终结论不能写成完整通过。

达到人工测试基本水平不是终点，而是进入更深度自动化发现的前提。只有先稳定发现人工测试会发现的基本问题，本工程再谈发现人工难以覆盖的特殊功能漏洞才有意义。
