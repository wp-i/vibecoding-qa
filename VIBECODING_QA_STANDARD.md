# vibeCoding 项目通用测试验收标准

本文档是交付给开发负责人的唯一标准文档。它定义 `agent-test` 面向所有 vibeCoding 项目的测试目标、最低验收门槛、必须产出的报告内容，以及当前已经实现的通用质量闸门。

## 1. 项目定位

`agent-test` 不是单项目测试脚本，也不是传统代码 review 工具。它面向 90% 以上由 AI 生成或低人工介入生成的 vibeCoding 项目，目标是以更低成本、更高效率产出接近人工测试人员的核心结果。

默认评估权重：

- 90%：功能是否符合需求文档、README、汇总 Markdown、示例输入输出和显式验收标准。
- 10%：明显交付风险，例如 key 明码、hard-code、固定样例过拟合、入口不可运行、Web UI 明显不可操作。

默认不重点评价代码是否优雅、结构是否清晰、抽象是否漂亮，也不做深度安全、深度健壮性或架构审美审查。除非这些问题直接导致需求不符合、功能不可用或明显交付事故。

## 2. 通用最低标准

所有 vibeCoding 项目都必须达到人工测试的基本产出水平。

这意味着测试不能只证明命令、页面或接口“跑过”，而必须检查用户基本流程和用户最终看到的结果。最终报告至少要包含：

- 测试用例。
- 复现流程。
- 实际结果。
- 期望结果。
- 证据。
- 严重程度。
- 开发人员可以直接定位和修复的缺陷描述。

如果没有做到这一点，最终报告不能写成完整功能验收通过。

## 3. 完整功能流程验收闸门

完整功能流程没有跑通，就等于没有完成该项目的功能验收。静态扫描、依赖检查、入口检查、help 命令、编译检查、单元测试或局部 smoke test 都只能作为辅助证据，不能替代用户真实主流程。

`agent-test` 必须优先尝试执行项目文档中最核心的端到端用户流程，例如：

- Web 项目：启动页面，模拟用户输入、点击、提交，并检查最终用户可见结果。
- CLI 项目：用 README 示例或代表性输入执行命令，并检查退出码、输出内容和生成文件。
- API 项目：启动服务，调用核心接口，并检查响应结构、状态码和用户/调用方可见结果。
- 报告/分析/推荐/生成类项目：执行一次完整任务，并检查最终报告、推荐、文件、截图或生成物是否有真实价值。

如果因为环境配置、外部依赖、账号、密钥、浏览器、数据库、第三方服务、人工登录、系统权限或网络条件等原因无法执行完整流程，报告必须明确标记为 `UNVERIFIED` 或 `PARTIAL`，并包含：

- 未能执行的完整流程名称。
- 阻塞原因，以及该原因是目标项目缺陷、测试环境缺失，还是需要用户提供的信息。
- 已经尝试过的命令或操作。
- 需要用户或测试人员补充的配置、账号、密钥、依赖、服务或手工步骤。
- 解除阻塞后的复测命令和验收标准。

如果阻塞不是项目本身的问题，报告不能把它伪装成项目功能失败；但也不能把项目写成通过。正确结论是“测试未完成/部分完成”，并协助用户解决环境问题后尽可能重新执行完整功能测试。

## 4. 用户可见输出质量闸门

只要被测项目会向用户生成报告、推荐、搜索结果、分析结论、任务列表、代码产物或其他用户可见输出，就必须检查该输出本身是否合格。

最低检查项：

- 输出是否真的回应了用户输入和需求文档，而不是只显示完成状态。
- 输出是否包含可操作结果、相邻参考、拒绝原因、搜索/执行覆盖说明或明确下一步。
- 输出是否是纯空白模板，例如只有“暂无”“未找到足够线索”“建议换关键词”。
- 输出中是否出现 debug、raw JSON、内部字段、栈信息、密钥名或工具流水。
- 输出是否存在明显重复段落、重复项目、重复 URL 或模板化堆叠。
- 如果输出给出项目、网站、仓库、文件、截图或链接，必须尽可能验证这些引用能打开且与用户需求相关。
- 如果输出包含评分、相关度、置信度或排序，明显为 0 分或无证据的项目不能作为正常参考展示。
- 对搜索、推荐、调研类项目，目录仓库、项目列表、newsletter、awesome list 这类泛化资源不能被当成直接满足需求的项目，除非用户明确要求这类资源。

没有执行这些检查时，最终功能验收报告必须标记为 `unverified`、`partial`，或明确说明缺少用户可见输出质量验收。

## 5. Web 和链接验证标准

如果项目存在 Web 前端、管理台、交互式页面或浏览器 UI，API 测试不能替代浏览器自动化测试。

最低浏览器验证内容：

- 启动真实页面。
- 打开页面并确认核心入口可见。
- 模拟用户输入、点击和提交。
- 等待关键 API 响应、结果区域出现、完成文本或错误文本。
- 记录 console error、page error、failed request。
- 保存截图或 DOM/bodyText 摘要作为证据。

如果用户报告中给出 GitHub 仓库、网站或其他外部引用，测试应尽可能打开这些链接，记录标题、页面信号和基础相关性判断。

## 6. AI 生成项目高频缺陷优先级

所有项目应优先排查 AI 生成项目的同质化问题：

- 假完成：流程显示完成，但结果没有价值。
- 空白交付：报告或页面只有空模板。
- 空壳功能：入口存在，但核心能力未实现。
- 固定样例伪通过：只对测试样例或文档样例有效。
- 过度过滤：候选存在或承诺 fallback，但最终输出被过滤成空。
- 无关推荐：返回项目、链接或内容与需求明显不相关。
- 不可复核：结论没有输入、输出片段、证据、链接或复现路径。
- 用户输出污染：debug、raw JSON、栈信息、内部字段暴露给用户。
- 评分/排序异常：0 分或无证据候选被展示为参考项目。

这些问题属于功能验收问题，应优先于代码结构、风格、抽象等传统工程审查项。

## 7. 必须交付的最终报告

每次正式产出必须包含两份 Markdown 报告和一份机器报告：

- `AGENT_TEST_QA_REPORT.md`：交付给开发人员、测试 agent 和修复 agent 的完整报告。
- `USER_QA_SUMMARY.md`：交付给非技术用户、产品或业务负责人的精简报告。
- `report.json`：交付给自动化系统、CI 和历史对比的机器可读报告。

开发/agent 报告必须包含：

- 总体结论：PASS/FAIL/PARTIAL/UNVERIFIED。
- 测试范围和边界。
- 执行过的命令、运行环境和 artifact 路径。
- 完整功能流程执行状态、阻塞原因和复测计划。
- 需求追踪矩阵。
- 测试用例矩阵。
- 缺陷列表。
- 每个缺陷的复现流程、实际结果、期望结果和证据。
- 用户可见输出质量检查结果。
- GitHub/外部 URL 基础核验结果。
- 修复验收标准。
- 残余风险和未覆盖项。

报告不能只给“建议”“观察”或“可能有问题”。对于功能性失败，必须能让开发人员立刻复现、排查和修复。

## 8. 当前已实现的通用能力

当前 `agent-test` 已实现以下通用能力：

- LLM-required acceptance profile。
- LLM 生成项目专属验收契约。
- AI 生成项目常见缺陷预检。
- 用户可见报告质量分析器：`src/analyzers/user-report-quality.js`。
- 用户可见输出质量闸门：`user-visible-report-quality-passed`。
- 用户报告校验 probe：`npm run probe:user-report`。
- GitHub URL 浏览器核验 probe：`npm run probe:github-url-review`。
- Playwright browser smoke probe：`npm run probe:browser`。
- 开发/agent 功能验收报告：`AGENT_TEST_QA_REPORT.md`。
- 非技术用户摘要报告：`USER_QA_SUMMARY.md`。
- 机器可读报告：`report.json`。
- 自举扫描：`npm run scan:self`。

## 9. 当前验证命令

项目自身验证命令：

```powershell
npm test
npm run scan:self
```

完整验证：

```powershell
npm run verify
```

浏览器运行时缺失时先执行：

```powershell
npm run setup:browser
```

## 10. 通过标准

一个 vibeCoding 项目的测试报告只有在满足以下条件时，才能视为达到基本验收标准：

- 已从 README/docs/汇总 Markdown 中抽取或确认核心需求。
- 已执行至少一个代表性完整功能流程；如果未执行，报告必须标记为 `UNVERIFIED` 或 `PARTIAL`，并列出阻塞原因和复测计划。
- 已执行与项目类型匹配的基本行为测试。
- 已检查用户最终可见输出。
- Web 项目已执行真实浏览器流程，或明确标记 Web 状态未验证。
- 用户输出中的链接、仓库或外部引用已尽可能核验。
- 已列出缺陷的复现步骤、实际结果、期望结果和证据。
- 报告明确区分已验证、未验证、风险、失败和测试平台自身限制。

如果缺少用户可见输出质量验收，报告不得标记为完整通过。

## 11. 核心判断

`agent-test` 的价值不是完全模拟人工测试人员的每一次操作，而是稳定实现同等或更好的测试产出。

第一目标：人工测试会发现的基本流程问题必须全部发现。

第二目标：测试人员常规测试流程覆盖的问题必须系统发现。

第三目标：在此基础上，进一步发现人工测试难以覆盖的特殊功能漏洞。
